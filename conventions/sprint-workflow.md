# Sprint Workflow

> **역할:** 팀이 스프린트를 어떻게 운영하는지에 대한 약속.

## 1. 스프린트 길이

- **1~2주 단위**로 유동적으로 운영하며, 각 `스프린트 플래닝` 시 이번 스프린트 길이를 정한다.

## 2. 스프린트 플래닝

> 스프린트 첫날, 이번 스프린트에 담을 작업을 정의한다.

- 플래닝에서는 **Story 단위까지만** 정의한다.
- Story는 사용자/시스템 관점의 검증 가능한 단위로 적는다. (feature 문서의 Story 정의와 동일)

```
사용자는 소셜 계정으로 로그인할 수 있다.
시스템은 업로드된 PDF의 유효성을 판정해 등록 여부를 결정한다.
```

- Story를 Jira 백로그에서 이번 스프린트로 가져온다.
- 구체적인 구현 작업(Task)은 플래닝에서 미리 쪼개지 않는다 → §3.

## 3. Task 운영

> Story를 실제 구현 작업으로 쪼갠 단위. **담당자가 직접** 만든다.

- 각 Story 아래에 **파트별 Task**를 만든다. Task명 앞에 파트 태그를 붙인다.

```
[FE] 로그인 화면 구현
[BE] 소셜 인증 API 연동
[AI] -
```

- **생성 시점:** 
	- 담당자가 작업에 **착수하는 시점**에 해당 Story 밑에 자신의 Task를 만든다.
- **작업 단위 = Task.** 
	- Git 브랜치 키(`YMC-XXX`)는 **이 Task의 이슈 키**를 쓴다.
  - 따라서 작업 전 Jira에 Task를 가장 먼저 등록한다.
	  - `Task`와 `Issue Num`이 존재해야 자동화(`브랜치 생성 → 진행 중` 등)가 동작한다.
  - 자동 상태 전환(`진행 중` / `Dev Done` / `Completely Done`)은 이 Task 이슈에 적용된다.
- **Story 완료 시점** 
  - 한 Story 밑 Task가 **전부** 끝나면 Story가 자동으로 따라 완료된다(롤업). <br>
  → `conventions/jira-workflow.md`

> 자동화가 깨지지 않으려면 순서가 중요하다: **Jira에서 Task 생성 → Task 키로 브랜치 생성 → 작업.**

## 4. 미확정 항목 (운영하며 결정)

> 아직 합의 전. 우선 각자 해보고 회고에서 방향을 정한다.

- **서브태스크:** 쓸지 말지 미정. **우선 각자 자율**로 시도해보고 결정.
	- 하나의 태스크가 1인에게 부여되는 환경이라 필요성이 크게 느껴지지 않음.
- **스토리 포인트(추정):** 도입 여부 미정. 당분간 생략하고, 필요성이 느껴지면 도입 검토.

## 5. 정기 회의

| 회의      | 시점           | 형태        | 비고             |
| ------- | ------------ | --------- | -------------- |
| 데일리 스크럼 | 매일 **13:00** | 온라인      | 짧게 진행 상황·이슈 공유 |
| 정기 회의   | 매일 **18:00** | 오프라인  | 필요 시 온라인/생략    |

## 6. 회고 (Retrospective)

- 매 스프린트 종료 시 **KPT** 방식으로 진행한다.
  - **Keep** — 잘된 것, 계속할 것
  - **Problem** — 문제였던 것
  - **Try** — 다음 스프린트에 시도할 개선
- 회고에서 나온 `Try`를 반영해 운영 방식을 업데이트한다.

## 7. 한눈에 보기

```
스프린트 시작
  └ 플래닝: Story 정의 (1~2주 분량)
        └ (스프린트 중) 담당자가 착수 시 [파트] Task 생성
              └ Task 키로 브랜치 → 커밋 → PR (자동 상태 전환)
매일 13:00 데일리 / 18:00 정기 회의(선택)
스프린트 종료
  └ KPT 회고 → 운영 방식 최적화
```
