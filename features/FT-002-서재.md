# Feature: FT-002 서재

## 1. Overview

- Summary: 로그인 사용자의 등록된 논문 목록을 서재 화면에 보여주고, 빈 상태·목록·등록 진입점·학습 진입을 제공한다.
- Status: In Progress
- Tracking: YMC-186 / `contracts/openapi.yaml` — `GET /api/papers` (목록), `GET /api/papers/{paperId}/status` (폴링)
- Depends on: FT-001 소셜 인증

## 2. Feature Boundary

### In Scope

- 서재 빈 상태 (등록 진입점 노출)
- 사용자별 논문 목록 조회 / 렌더 (WF-016)
- 논문 행 상태 표시 (진행 중 / 완료 / 실패)
- 완료 1회 표시 seen 상태 기록·반영 (서버 영속)
- 논문 선택 → 학습 페이지(FT-004) 진입

### Out of Scope

- 논문 등록 / 분석 → FT-003
- 분석 실패 행의 재시도 / 복구 액션 → FT-003(후속) (서재는 실패 상태 표시까지만)
- 논문 검색 / 정렬 / 필터, 폴더 · 태그 분류 → 후속(post-MVP)
- 논문 삭제 / 이름 변경 / 메타 편집 → 후속(post-MVP)
- 다운로드 후 재분석 / 파일 교체 → 후속(post-MVP)

## 4. Stories

### Story 1. 사용자는 빈 서재에서 첫 논문 등록을 시작할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 1-2
- Acceptance Criteria:
  - 등록된 논문이 없는 사용자는 빈 서재(WF-004)와 등록 진입점을 본다.
  - 등록 진입점 선택 시 FT-003의 등록 모달로 전환된다.
- Depends on: -

### Story 2. 사용자는 등록한 논문 목록을 서재에서 볼 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 5
- Acceptance Criteria:
  - 로그인한 사용자 본인 소유 논문만 목록(WF-016)에 표시된다.
  - 각 논문 행이 상태(진행 중/완료/실패)와 함께 표시된다.
- Depends on: FT-001 Story 2

### Story 3. 사용자는 논문의 분석 진행·완료·실패 상태를 서재에서 볼 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 4-5
- Tracking: YMC-186
- Acceptance Criteria:
  - 업로드 직후 논문이 진행 중 상태(WF-006)로 서재에 표시된다.
  - 클라이언트는 `GET /api/papers/{paperId}/status`를 **폴링**해 행 상태를 갱신
  - 계약의 `PaperStatus` 6개 값을 아래 3개 표시 상태로 매핑한다.

    | `PaperStatus` (계약) | 서재 행 표시 | 비고 |
    |---|---|---|
    | `UPLOAD_PENDING` | 진행 중 | FE가 S3 업로드 중 |
    | `UPLOADED` | 진행 중 | 파싱 발행 직전의 짧은 전이 상태 |
    | `PROCESSING` | 진행 중 | 파싱 진행 중 |
    | `COMPLETED` | 완료 (WF-007) | 학습 페이지 진입 가능 |
    | `FAILED` | 실패 (WF 미정) | 파일은 올라갔으나 **파싱이 깨진** 경우 |
    | `EXPIRED` | 실패 (WF 미정) | 업로드가 끝내 확인되지 않아 **파싱을 시작조차 못 한** 경우. 원인은 `FAILED`와 다르지만 **사용자에게는 똑같이 '실패'로 보인다.** MVP에서는 발생하지 않는다.

  - 분석이 완료되면 행이 완료 상태(WF-007)로 갱신되고, '완료' 전이 표시는 1회만 노출된다(서버 seen 기록 기반 — 한 번 본 뒤 기기·세션과 무관하게 무표시).
  - 분석이 실패하면 행이 실패 상태(WF 미정)로 표시된다. 실패 행의 재시도·복구 액션은 이 feature 밖(FT-003 후속).
  - **MVP는 "무한 PROCESSING" 행을 허용.** (파싱 워커가 결과를 돌려주지 않는 경우) FE가 임의로 타임아웃을 걸어 '실패'로 표시하지 않는다(상태 판정은 BE 소유). (ADR-001 §5)
  - 완료 상태가 아닌 행은 **비활성 상태로 표시**되어 선택할 수 없다 (학습 페이지 진입 불가).
- Depends on: Story 2, FT-003 Story 6

### Story 4. 사용자는 서재에서 논문을 선택해 학습 페이지로 들어갈 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 5
- Acceptance Criteria:
  - `COMPLETED` 상태 논문 행을 선택하면 해당 논문의 학습 페이지 초기 상태(WF-008)로 이동한다.
- Depends on: FT-004 Story 1

### Story 5. 사용자는 완료된 논문의 원본 PDF를 다운로드할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 5
- Acceptance Criteria:
  - `COMPLETED` 상태 논문 행에서 원본 PDF 다운로드를 제공한다.
  - `PROCESSING`/`FAILED` 행에는 다운로드를 제공하지 않는다.
  - 저장되는 파일명은 등록 시 원본 파일명(`filename`)을 따른다(파싱이 추출한 제목이 아님).
  - 다운로드는 BE가 발급한 presigned GET URL로 브라우저가 S3에서 직접 받는다(ADR-001 대칭). 파일 바이트는 BE를 거치지 않는다.
- Depends on: Story 3

## 5. Done Criteria

- 빈 상태 → 등록 진입 → 목록 표시 → 학습 진입까지 서재 흐름이 동작한다.
- 사용자별 논문만 노출되고, 행 상태가 분석 진행에 따라 갱신된다.

## 6. Open Questions

- 목록 정렬·페이지네이션 기준 (최근 등록순? 무한 스크롤?) — MVP 계약은 정렬·페이징 없는 단순 목록으로 시작한다.
- 실패 상태 행의 표시 방식(아이콘/메시지)과 와이어프레임이 아직 없음(WF 미정). MVP는 "표시만" 범위 — **와이어프레임 추가 필요.**
- 폴링 주기를 얼마로 할 것인가? (FE 결정 사항)
- 완료 1회 표시용 `seen` 기록 API가 아직 계약에 없다 — FT-002 본격 스프린트에서 `contracts/openapi.yaml`에 추가한다.

> **Resolved** — 상태 갱신 전달 방식: FT-003의 결정(**폴링**)을 따른다.
>
> **Resolved** — 상태값 매핑: 계약의 `PaperStatus` 6개 값 → 표시 3개(진행 중/완료/실패)의 매핑을 Story 3에 표로 확정했다.
>
> **Resolved** — `EXPIRED` 표시: 사용자에게는 `FAILED`와 **똑같이 '실패'로** 보여준다. 단, **BE 상태값으로는 `EXPIRED`를 남긴다.**

> **Resolved** — 완료 1회 표시: 화면 한정이 아니라 **서버에 seen 상태를 영속**한다(사용자·논문 단위). 한 번 본 뒤에는 기기·세션과 무관하게 다시 표시되지 않는다. 완료 표시를 본 시점에 seen으로 기록하는 작은 서버 write가 FT-002 책임에 포함된다.
