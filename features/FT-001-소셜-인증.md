# Feature: FT-001 소셜 인증

## 1. Overview

- Summary: 사용자가 소셜 계정(MVP: Google)으로 회원가입/로그인하고 인증 세션을 유지한다.
- Status: In Progress
- Tracking: YMC-211 (에픽) — YMC-213(BE)·YMC-214(FE)·YMC-215(업로드 연동)
- Depends on: -

## 2. Feature Boundary

### In Scope

- 소셜 로그인 / 회원가입 (MVP: Google)
- 인증 모달 (랜딩 위 오버레이) 진입 / 닫기
- 최초 로그인 시 사용자 레코드 생성, 기존 사용자 식별
- 인증 세션 발급 / 유지 / 로그아웃
- 인증 성공 후 서재 라우팅


### Out of Scope

- Kakao · Naver 로그인 → 후속(post-MVP)
- 이메일 / 비밀번호 로그인
- 프로필 편집, 회원 탈퇴
- 권한 / 역할(Role) 기반 접근 제어
- 소셜 인증 실패 / 취소 복구 흐름 (별도 Exception flow 후보)

## 4. Stories

### Story 1. 사용자는 소셜 계정(Google·Kakao·Naver)으로 로그인·회원가입할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-001 Step 1-4
- Acceptance Criteria:
  - 랜딩(WF-001)에서 로그인 / 회원가입 선택 시 인증 모달(WF-003)이 열린다.
  - Google로 인증을 완료하면 모달이 닫히고 서재(WF-004)로 이동한다. (MVP: Google만 지원)
  - 모달 닫기 선택 시 비로그인 랜딩(WF-001)으로 돌아간다.
- Depends on: -

### Story 2. 시스템은 인증 결과로 사용자를 식별하고 세션을 발급한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-001 Step 3-4
- Acceptance Criteria:
  - 클라이언트가 선택한 provider 인증을 서버에 요청하면, 서버가 인증을 처리해 로그인 결과를 반환한다. (client → BE)
  - `provider + provider_id`로 기존 사용자를 식별하고, 신규면 레코드를 생성한다.
  - 동일 사용자가 서로 다른 provider로 로그인하면 MVP 기준 **별개 계정**으로 취급한다(계정 병합 없음).
  - 인증 성공 시 세션(토큰/쿠키)을 발급한다.
- Depends on: -

### Story 3. 사용자는 세션이 유지된 채 재방문하고, 원할 때 로그아웃할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-001 Step 4
- Acceptance Criteria:
  - 유효한 세션이 있는 사용자가 랜딩 진입 시 로그인 상태(WF-002)가 노출된다.
  - 로그아웃 시 세션이 무효화되고 비로그인 랜딩(WF-001)으로 전환된다.
- Depends on: Story 2

## 5. Done Criteria

- Google로 신규 가입 · 재로그인 · 로그아웃이 동작한다.
- 인증 성공 후 서재 진입, 세션 유지 / 만료 동작이 확인된다.
