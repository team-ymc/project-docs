# Feature: FT-001 소셜 인증

## 1. Overview

- Summary: 사용자가 Google·Kakao·Naver 소셜 계정으로 회원가입/로그인하고 인증 세션을 유지한다.
- Status: Draft
- Tracking: -
- Depends on: -

## 2. Feature Boundary

### TASK In Scope

- 소셜 로그인 / 회원가입 (Google, Kakao, Naver)
- 인증 모달 (랜딩 위 오버레이) 진입 / 닫기
- 최초 로그인 시 사용자 레코드 생성, 기존 사용자 식별
- 인증 세션 발급 / 유지 / 로그아웃
- 인증 성공 후 서재 라우팅

#### DRAFT
  - 랜딩(WF-001/002)에서 로그인·회원가입 버튼 → 인증 모달(WF-003) open/close
  - Google / Kakao / Naver OAuth 시작 → IdP 리다이렉트
  - OAuth 콜백 수신 → 토큰·프로필 검증
  - provider + provider_id 로 기존 사용자 식별, 없으면 신규 생성
  - 세션(토큰/쿠키) 발급, 만료/갱신, 로그아웃 시 무효화
  - 인증 성공 시 빈 서재(WF-004)로 이동, 기존 사용자는 서재 목록으로
  - 로그인 사용자가 랜딩 진입 시 로그인 상태(WF-002) 노출

## 4. Stories

### Story 1. 소셜 계정으로 로그인 / 회원가입 `MVP`

- Story: 비로그인 사용자로서, 소셜 계정으로 가입/로그인해 바로 내 서재로 들어가고 싶다.
- type: USER
- Source Userflows: UF-001 Step 1-4
- Acceptance Criteria:
  - 랜딩(WF-001)에서 로그인 / 회원가입 선택 시 인증 모달(WF-003)이 열린다.
  - Google / Kakao / Naver 중 하나로 인증을 완료하면 모달이 닫히고 빈 서재(WF-004)로 이동한다.
  - 모달 닫기 선택 시 비로그인 랜딩(WF-001)으로 돌아간다.
- Depends on: -

### Story 2. BE → 소셜 IdP 인증 검증 `MVP`

- Story: 시스템으로서, 소셜 IdP에 인증 코드를 교환·검증 요청해 받은 프로필로 로그인 처리하고 세션을 발급한다.
- type: SYSTEM
- Source Userflows: UF-001 Step 3-4
- Acceptance Criteria:
  - IdP 콜백의 인증 코드를 토큰으로 교환하고 프로필을 조회·검증한다. (BE → IdP)
  - `provider + provider_id`로 기존 사용자를 식별하고, 신규면 레코드 생성·기존이면 매핑 계정으로 로그인 처리한다.
  - 동일 사용자가 서로 다른 provider로 로그인하면 MVP 기준 **별개 계정**으로 취급한다(계정 병합 없음).
  - 인증 성공 시 세션(토큰/쿠키)을 발급한다.
- Depends on: -

### Story 3. 인증 세션 유지 및 로그아웃 `MVP`

- Story: 로그인 사용자로서, 재방문 시 세션이 유지되고 원할 때 로그아웃하고 싶다.
- type: USER
- Source Userflows: UF-001 Step 4
- Acceptance Criteria:
  - 유효한 세션이 있는 사용자가 랜딩 진입 시 로그인 상태(WF-002)가 노출된다.
  - 로그아웃 시 세션이 무효화되고 비로그인 랜딩(WF-001)으로 전환된다.
- Depends on: Story 2

## 5. Done Criteria

- 세 개 provider 모두로 신규 가입 · 재로그인 · 로그아웃이 동작한다.
- 인증 성공 후 서재 진입, 세션 유지 / 만료 동작이 확인된다.
