# Feature: FT-003 논문 등록 · 분석

## 1. Overview

- Summary: 사용자가 PDF를 업로드하면 검증을 거쳐 서재에 등록되고, 분석이 끝나면 학습 가능한 상태가 된다.
- Status: In Progress
- Depends on: FT-002 서재
- Tracking: YMC-179 (Epic) / `contracts/frontend-backend/openapi.yaml` (FE↔BE) · `contracts/backend-ai/openapi.yml` (BE↔AI)

## 2. Feature Boundary

### In Scope

- PDF 업로드로 논문 등록 (등록 모달)
- **파일명/논문 중복 판정** 후 논문 항목 생성 (내용 기반 유효성 판정은 MVP 범위 밖 — 아래 Resolved)
- 문서 파싱 파이프라인(도메인 중립): 원본 문서를 파싱해 **Markdown(.md) 본문 + 분리된 미디어 파일(그림 · 표 · 수식 등)**로 구조화
- 분석 완료 시 `COMPLETED` 전이 및 서재 반영, 분석 실패 시 `FAILED` 전이

### Out of Scope

- URL 입력 방식 논문 등록 → 후속(post-MVP) 
- **내용 기반 유효성 판정**(논문 메타데이터 존재 여부, 페이지 수 제한 등) → 후속(post-MVP). 아래 Resolved 참고
- 파일명 외 정교한 중복 판정(내용 해시, 메타데이터 기반) → 후속(post-MVP)
- **분석 실패의 복구 흐름(재시도 · 재업로드 등) → 후속(post-MVP).** 상태를 `FAILED`로 전이시키고 조회 가능하게 하는 것까지가 이 feature의 책임
- **정체 레코드 정리(reconciliation batch) → 후속(post-MVP).** MVP는 정체를 방치한다 (ADR-001 §5)
- 개념/미디어 위키 생성 → FT-008 위키 생성
- 구조 맵(개념 연결·중요요소 인덱스) 생성 → FT-009 구조 맵

## 4. Stories

### Story 1. 사용자는 PDF를 업로드해 논문을 등록할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 1-3
- Acceptance Criteria:
  - 서재 빈 상태(WF-004)에서 등록 버튼 선택 시 등록 모달(WF-005)이 열린다.
  - 로그인 사용자가 PDF를 선택해 업로드를 실행하면 모달이 닫힌다.
  - 파일명이 중복이 아니면 논문이 저장되어 서재에 항목으로 추가된다(WF-006).
- Depends on: FT-002 Story 1

### Story 2. 시스템은 파일명 중복을 판정해 등록 여부를 결정한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 3-4
- Acceptance Criteria:
  - `POST /api/papers` 시점에 사용자의 기존 논문과 **파일명 1:1 매핑**으로 중복을 판정한다.(MVP)
  - 중복이면 레코드를 만들지 않고 `409 DUPLICATE_FILENAME`을 반환한다.
  - 중복이 아니면 레코드를 `UPLOAD_PENDING`으로 생성하고 presigned URL을 발급한다.
- Depends on: -

### Story 3. 시스템은 업로드 완료를 확인하고 파싱을 요청한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 4
- Tracking: YMC-182
- Acceptance Criteria:
  - `POST /api/papers/{paperId}/complete` 수신 시 S3 HEAD로 객체 존재를 확인하고 `UPLOAD_PENDING → UPLOADED`로 전이한다.
  - 전이는 조건부로 수행해 **중복 처리를 방지한다**
  - 파싱 요청을 큐에 발행하고 `UPLOADED → PROCESSING`으로 전이한다. 메시지에는 `paperId`와 `fileKey` 참조를 담는다.
  - 객체가 없으면 상태를 전이시키지 않고 4xx를 반환한다.
  - 사용자는 파싱을 기다리지 않는다
- Depends on: Story 2

### Story 4. 시스템은 문서를 .md 본문 + 미디어로 구조화한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 4-5
- Tracking: YMC-189
- Owner: AI 파싱 워커
- Acceptance Criteria:
  - 파싱 서버는 `fileKey`로 S3에서 원본을 직접 읽고, 산출물을 S3에 쓴다. (ADR-001)
  - 본문을 섹션 · 문단 단위의 **Markdown(.md)**으로 구조화하고, 그림 · 표 · 수식은 **별도 미디어 파일로 분리**해 본문에서 참조한다.
  - 산출물(**.md 본문 + 미디어 파일 세트**)은 FT-004 Reader 렌더링과 FT-008 · FT-009 생성의 입력으로 쓸 수 있는 형태다.
  - 성공·실패 **모두** `paperId`와 처리 결과를 포함해 결과 큐에 발행한다.
  - 파싱 서버는 논문 처리 상태는 직접 쓰지 않는다. (상태 전이는 BE 책임 - Story 5)
- Depends on: Story 3

### Story 5. 시스템은 파싱 결과를 수신해 완료·실패 상태로 전이한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 5
- Tracking: YMC-183
- Acceptance Criteria:
  - 파싱 결과를 수신해 `PROCESSING → COMPLETED` 또는 `PROCESSING → FAILED`로 전이한다.
  - 실패 시 실패 사유(`error.code`)를 기록한다. MVP에서 사용자에게 code를 그대로 노출하지는 않는다.
  - **상태의 단일 writer는 BE다.**
  - 결과가 끝내 오지 않으면(워커 유실 등) 레코드는 `PROCESSING`에 남는다 — **MVP는 정리하지 않는다.** (ADR-001 §5)
- Depends on: Story 4

### Story 6. 시스템은 현재 처리 상태를 클라이언트가 조회할 수 있게 제공한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 4-5
- Tracking: YMC-184
- Acceptance Criteria:
  - 클라이언트는 `GET /api/papers/{paperId}/status`로 현재 상태를 **폴링**해 조회할 수 있다. (아래 Resolved — 상태 전달 방식)
  - 응답 상태값은 `contracts/frontend-backend/openapi.yaml`의 `PaperStatus` enum을 따른다: `UPLOAD_PENDING · UPLOADED · PROCESSING · COMPLETED · FAILED · EXPIRED`.
  - 존재하지 않는 `paperId`는 404와 `PAPER_NOT_FOUND` 에러 코드를 반환한다.
- Depends on: Story 1 

### Story 7. 시스템은 동일 파일의 기본 파싱 결과를 재사용한다 `post-MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 3-5
- Related: ADR-003
- Acceptance Criteria:
  - 완전히 동일한 파일이 다른 파일명 또는 다른 사용자에 의해 반복 등록되면 기본 파싱을 중복해서 요청하지 않는다.
  - 동일 파일의 판정 기준은 S3가 실제 업로드 바이트와 일치한다고 검증한 전체 파일 SHA-256이다.
  - 각 사용자는 자신의 `Paper`를 가지며, 동일 파일의 `Paper`들은 하나의 내용 단위와 기본 파싱 결과를 공유한다.
  - 기존 기본 파싱이 완료됐다면 새 `Paper`는 기존 결과를 참조해 `COMPLETED`가 된다.
  - 기존 기본 파싱이 진행 중이라면 새 `Paper`는 같은 작업의 완료를 기다리며 `PROCESSING`으로 조회된다.
  - 손상 PDF 등 콘텐츠 자체의 영구 실패는 같은 파일을 기다리는 모든 `Paper`에 반영한다.
  - 네트워크·외부 API timeout 등 일시적 실패는 재시도하고, 최대 시도 횟수를 초과하면 최종 실패로 처리한다.
  - DOI·arXiv ID·제목·저자 등을 이용한 의미적 동일 논문 판정은 후속 Story로 분리한다.
- Depends on: Story 3-5

## 5. Done Criteria

- PDF 업로드 → 중복 판정 → 저장 → 분석 → 서재 완료 상태까지 end-to-end로 동작한다.
- 파싱이 **결과를 돌려준** 경우, 실패면 항목이 `FAILED`로 전이되어 조회된다.
- 완료된 논문이 FT-004 Reader에서 본문으로 열린다.

## 6. Open Questions

- 그림·표·수식을 *추출*은 하되 *분석(의미 해석)*까지 할 것인가? — 논문/E-Book은 대개 그 아래에 설명글이 있어 중복일 수 있음. (추출은 In Scope 확정, 분석 여부만 미정)
- 서버 토폴로지: 문서 처리(중복·유효성·파싱)를 Agent AI 서버(LLM 작업: 번역·설명·튜터)와 별도 서비스로 분리할 것인가? (main BE 포함 분리 범위 — 결정되면 각 seam을 contracts로)
- 파싱 데이터 저장소: 구조화 본문을 NoSQL vs PostgreSQL 중 어디에 저장하는가?
- 제품 도메인 범위: '논문' 등록은 MVP 첫 검증 대상일 뿐, 장기적으로 학습형 E-Book 등 일반 문서로 확장될 수 있다. 파싱 파이프라인(원본→.md+미디어)은 도메인 중립으로 두고, 논문-특화 가정(메타 유효성 판정 등)만 교체 가능한 도메인 레이어로 둔다. (theme 확정 시 이 feature의 등록·검증 슬롯만 영향)

> **Resolved** — 유효성·중복 판정의 위치: MVP는 **파일명 중복만** BE가 `POST /api/papers` 시점에 판정한다. **내용 기반 유효성 판정과 정교한 중복 판정은 MVP에서 하지 않는다.** 정확한 파일 바이트의 중복 제거와 기본 파싱 결과 공유는 post-MVP Story 7 및 ADR-003에서 다룬다.
>
> 이유: presigned URL 방식(ADR-001)에서 BE는 파일 바이트를 볼 수 없다. 등록(레코드 생성)이 업로드보다 **먼저** 일어나므로 "판정 후 등록"이라는 초기 초안은 물리적으로 성립하지 않는다.
>
> 후속: Story 7에서는 FE가 계산하고 S3가 검증한 전체 파일 SHA-256으로 정확히 동일한 파일을 판정한다. BE는 사용자별 `Paper`와 내용 단위의 연결 및 기본 파싱 준비 상태를 관리한다. DOI·제목·저자 기반 의미적 동일 논문 판정과 내용 유효성 검사는 AI 서버 또는 별도 판정 서버의 후속 책임으로 분리하고, 각 seam은 contracts로 정의한다.

> **Resolved** — 상태 전달 방식: **폴링**으로 한다. 클라이언트는 `GET /api/papers/{paperId}/status`를 주기 호출해 상태를 반영한다. SSE는 MVP에서 채택하지 않는다(연결 유지·재접속·스케일 비용 대비 이득이 작음). 이 결정은 FT-003가 소유하고 FT-002가 따른다. 계약: `contracts/frontend-backend/openapi.yaml`.
>
> **Resolved** — 상태 표시 범위: 진행 중 · 완료 · **실패** 표시는 모두 **MVP In Scope**다. (초기 초안은 "완료 반영만"이었으나, FT-002 Story 3 · UF-002 Step 4 · WF-006 · openapi의 `PROCESSING`/`FAILED`가 모두 표시를 전제하므로 정정한다.) 후속으로 미루는 것은 실패의 **복구 흐름(재시도 등)**뿐이다.
>
> **Resolved** — 상태 용어: 문서 전반에서 `ready` 대신 계약의 enum 값 `COMPLETED`를 쓴다. 상태값의 SSOT는 `contracts/frontend-backend/openapi.yaml`의 `PaperStatus`다.
>
> **Resolved** — 파싱 산출물 포맷: 파이프라인은 원본 문서를 **Markdown(.md) 본문 + 분리된 미디어 파일**로 출력한다. 이 .md/미디어 분리 파이프라인을 **가장 먼저** 구축하며, 도메인 중립으로 설계한다.
>
> **Resolved** — 파싱 엔진: MVP는 **상용 파싱 API**로 시작하고, 추후 오픈소스 엔진으로 전환한다. BE↔파싱서버 contracts는 유지한 채 구현체만 교체한다(엔진 교체가 계약을 안 깨는 게 seam 분리의 목적).
>
> **Resolved** — AI Wiki화 경계: 위키 생성은 이 feature에 포함하지 않는다. `COMPLETED`는 **구조 추출 완료**까지이며, 위키 생성은 FT-008, 구조 맵은 FT-009로 분리한다. (후속 AI 처리의 큐/MQ 도입은 해당 feature에서 다룸)
