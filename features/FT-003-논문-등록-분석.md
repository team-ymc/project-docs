# Feature: FT-003 논문 등록 · 분석

## 1. Overview

- Summary: 사용자가 PDF를 업로드하면 검증·저장 후 비동기 분석을 거쳐 학습 가능한 상태로 서재에 등록한다.
- Status: Draft
- Tracking: -
- Depends on: FT-002 Library

## 2. Feature Boundary

### TASK In Scope

- PDF 업로드로 논문 등록 (등록 모달)
- 업로드 파일 검증 (유효성 · 중복) 후 논문 항목 생성 (진행 상태)
- 비동기 논문 분석 파이프라인 (텍스트 · 그림 · 표 · 수식 추출 · 구조화)
- 분석 진행 → 완료 상태 전이 및 서재 반영
- 이 feature의 `ready` 경계 = **구조 추출 완료까지**. 위키 생성(FT-008)·구조 맵(FT-009)은 이 산출물 위에 얹히는 별도 feature

#### DRAFT
  - ③ 구현 절차 스크래치 (스프린트 시작 시 BE/AI 트랙으로 distill)
  - 업로드 API → 인증/권한 확인 → main BE 파일 검증
  - 유효성·중복 확인 후 스토리지(S3 가정) 원본 저장 + 원본 경로 DB 기록
  - 파싱 요청 → 텍스트 추출 · 구조화 → 파싱 결과 DB 기록
  - BE는 파싱 완료/실패 + 저장 위치만 수신 → 사용자 상태 갱신
  - 완료 후 AI 가공(Wiki화) 처리 큐로 비동기 전달 → 진행 상태 주기 갱신 → 완료 시 DB 기록
  - 산출물(구조화 본문)은 FT-004 Reader가 렌더링에 사용

## 4. Stories

### Story 1. PDF 업로드로 논문 등록 `MVP`

- Story: 로그인 사용자로서, 빈 서재에서 PDF를 올려 첫 논문을 등록하고 싶다.
- type: USER
- Source Userflows: UF-002 Step 1-3
- Acceptance Criteria:
  - 서재 빈 상태(WF-004)에서 등록 버튼 선택 시 등록 모달(WF-005)이 열린다.
  - 인증된 사용자가 PDF를 선택해 업로드를 실행하면 모달이 닫힌다.
  - 검증을 통과한 업로드는 원본이 저장되고 `status=processing` 논문 레코드가 생성되어, 서재에 진행 중 행(WF-006)으로 나타난다.
- Depends on: FT-002 Story 1

### Story 2. BE → 유효성·파싱 서버: 유효성·중복 검사 요청 `MVP`

- Story: 시스템으로서, 업로드된 PDF의 유효성·중복 판정을 유효성·파싱 서버에 요청해 등록 가능 여부를 받는다.
- type: SYSTEM
- Source Userflows: UF-002 Step 3-4
- Acceptance Criteria:
  - BE가 업로드 PDF를 유효성·파싱 서버에 검사 요청한다. (BE → 파싱 서버)
  - 유효성 판정 기준(MVP): 첫 n페이지에 논문 메타데이터 존재 + 전체 m페이지 이내.
  - 중복 판정(MVP): 사용자 기존 논문과 **파일명 1:1 매핑**. 동일 항목이면 등록 대신 사용자에게 알린다.
  - 검사 결과(유효/무효/중복)를 수신해 등록 진행 여부를 결정한다.
- Depends on: -

### Story 3. BE → 유효성·파싱 서버: 전체 문서 파싱 요청 `MVP`

- Story: 시스템으로서, 검증된 논문의 전체 파싱을 유효성·파싱 서버에 비동기로 요청해 구조화 산출물을 받는다.
- type: SYSTEM
- Source Userflows: UF-002 Step 4-5
- Acceptance Criteria:
  - BE가 파싱 서버에 전체 문서 파싱을 요청한다 — main BE 요청-응답을 점유하지 않는 비동기 방식. (BE → 파싱 서버)
  - 파싱 서버는 텍스트·그림·표·수식을 추출하고 섹션·문단 단위로 구조화한다.
  - BE는 파싱 완료/실패와 산출물 저장 위치를 수신하고, 완료 시 논문 상태를 `processing → ready`로 전이한다.
  - 산출물은 FT-004 Reader 렌더링과 FT-008 Wiki·FT-009 구조 맵 생성의 입력으로 쓸 수 있는 형태다.
- Depends on: Story 2

## 5. Done Criteria

- PDF 업로드 → 검증 → 저장 → 비동기 분석 → 서재 완료 상태까지 end-to-end로 동작한다.
- 완료된 논문이 FT-004 Reader에서 본문으로 열린다.

## 6. Open Questions

- 파싱 데이터 저장소: 구조화 본문을 NoSQL vs PostgreSQL 중 어디에 저장하는가?
- 상태 전달 방식: 진행/완료 상태를 polling vs SSE 중 무엇으로 클라이언트에 전달하는가?
