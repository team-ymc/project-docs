# Feature: FT-003 논문 등록 · 분석

## 1. Overview

- Summary: 사용자가 PDF를 업로드하면 검증을 거쳐 서재에 등록되고, 분석이 끝나면 학습 가능한 상태가 된다.
- Status: Draft
- Tracking: -
- Depends on: FT-002 서재

## 2. Feature Boundary

### In Scope

- PDF 업로드로 논문 등록 (등록 모달)
- 업로드 파일 검증 (유효성 · 중복) 후 논문 항목 생성
- 문서 파싱 파이프라인(도메인 중립): 원본 문서를 파싱해 **Markdown(.md) 본문 + 분리된 미디어 파일(그림 · 표 · 수식 등)**로 구조화
- 분석 완료 시 ready 상태 전이 및 서재 반영

### Out of Scope

- URL 입력 방식 논문 등록 → 후속(post-MVP) 
- 파일명 외 정교한 중복 판정(내용 해시, 메타 데이터 등 기반) → 후속(post-MVP)
- 논문 분석 실패 시 사용자 대응 및 복구 흐름(재시도 등) → 후속(post-MVP)
- 분석 진행 상태(처리 중)를 사용자에게 표시·알림 → 후속(post-MVP). MVP는 완료(ready) 반영만 한다
- 개념/미디어 위키 생성 → FT-008 위키 생성
- 구조 맵(개념 연결·중요요소 인덱스) 생성 → FT-009 구조 맵

## 4. Stories

### Story 1. 사용자는 PDF를 업로드해 논문을 등록할 수 있다 `MVP`

- type: USER
- Source Userflows: UF-002 Step 1-3
- Acceptance Criteria:
  - 서재 빈 상태(WF-004)에서 등록 버튼 선택 시 등록 모달(WF-005)이 열린다.
  - 로그인 사용자가 PDF를 선택해 업로드를 실행하면 모달이 닫힌다.
  - 검증을 통과하면 논문이 저장되어 서재에 항목으로 추가된다(WF-006).
- Depends on: FT-002 Story 1

### Story 2. 시스템은 업로드된 PDF의 유효성·중복을 판정해 등록 여부를 결정한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 3-4
- Acceptance Criteria:
  - 유효성 판정 기준(MVP): 첫 n페이지에 논문 메타데이터 존재 + 전체 m페이지 이내.
  - 중복 판정(MVP): 사용자 기존 논문과 **파일명 1:1 매핑**. 동일 항목이면 등록 대신 사용자에게 알린다.
  - 판정 결과(유효 / 무효 / 중복)에 따라 등록 진행 여부가 결정된다.
- Depends on: -

### Story 3. 시스템은 검증된 문서를 파싱해 .md 본문 + 미디어로 구조화하고, 완료·실패 상태로 전이한다 `MVP`

- type: SYSTEM
- Source Userflows: UF-002 Step 4-5
- Acceptance Criteria:
  - 검증을 통과한 문서는 비동기로 파싱에 들어가며, 그동안 사용자는 기다리지 않는다.
  - 파싱은 본문을 섹션 · 문단 단위의 **Markdown(.md)**으로 구조화하고, 그림 · 표 · 수식은 **별도 미디어 파일로 분리**해 본문에서 참조한다.
  - 파싱이 끝나면 해당 항목이 완료(ready)로 전이되어 서재에서 학습 가능해진다. (진행 상태 표시는 Out of Scope — 완료만 반영)
  - 산출물(**.md 본문 + 미디어 파일 세트**)은 FT-004 Reader 렌더링과 FT-008 · FT-009 생성의 입력으로 쓸 수 있는 형태다.
- Depends on: Story 2

## 5. Done Criteria

- PDF 업로드 → 검증 → 저장 → 분석 → 서재 완료 상태까지 end-to-end로 동작한다.
- 완료된 논문이 FT-004 Reader에서 본문으로 열린다.

## 6. Open Questions

- 그림·표·수식을 *추출*은 하되 *분석(의미 해석)*까지 할 것인가? — 논문/E-Book은 대개 그 아래에 설명글이 있어 중복일 수 있음. (추출은 In Scope 확정, 분석 여부만 미정)
- 서버 토폴로지: 문서 처리(중복·유효성·파싱)를 Agent AI 서버(LLM 작업: 번역·설명·튜터)와 별도 서비스로 분리할 것인가? (main BE 포함 분리 범위 — 결정되면 각 seam을 contracts로)
- 파싱 데이터 저장소: 구조화 본문을 NoSQL vs PostgreSQL 중 어디에 저장하는가?
- 상태 전달 방식: 완료(ready) 전이를 polling vs SSE 중 무엇으로 클라이언트에 전달하는가? (진행 상태 표시는 Out of Scope라 MVP에선 완료 반영만 대상. 이 결정은 FT-003가 소유하고 FT-002가 따른다)
- 제품 도메인 범위: '논문' 등록은 MVP 첫 검증 대상일 뿐, 장기적으로 학습형 E-Book 등 일반 문서로 확장될 수 있다. 파싱 파이프라인(원본→.md+미디어)은 도메인 중립으로 두고, 논문-특화 가정(메타 유효성 판정 등)만 교체 가능한 도메인 레이어로 둔다. (theme 확정 시 이 feature의 등록·검증 슬롯만 영향)

> **Resolved** — 파싱 산출물 포맷: 파이프라인은 원본 문서를 **Markdown(.md) 본문 + 분리된 미디어 파일**로 출력한다. 이 .md/미디어 분리 파이프라인을 **가장 먼저** 구축하며, 도메인 중립으로 설계한다.
>
> **Resolved** — 파싱 엔진: MVP는 **상용 파싱 API**로 시작하고, 추후 오픈소스 엔진으로 전환한다. BE↔파싱서버 contracts는 유지한 채 구현체만 교체한다(엔진 교체가 계약을 안 깨는 게 seam 분리의 목적).
>
> **Resolved** — AI Wiki화 경계: 위키 생성은 이 feature에 포함하지 않는다. `ready`는 **구조 추출 완료**까지이며, 위키 생성은 FT-008, 구조 맵은 FT-009로 분리한다. (후속 AI 처리의 큐/MQ 도입은 해당 feature에서 다룸)
